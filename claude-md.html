CLAUDE.md
Best Practices

CLAUDE.md is automatically loaded by Claude at the start of every session — it's your project's instruction manual. Write it well and save hours of repeated prompting.

Overview

What is CLAUDE.md?

A markdown file in your project root that Claude reads automatically at session start. Think of it as a README for Claude — not for humans.

🤖
Auto-Loaded Every Session
Claude reads CLAUDE.md before your first message. No need to paste instructions repeatedly.
📂
Multiple Locations
Home folder, project root, and subdirectories — all CLAUDE.md files are merged.
🔗
Supports @imports
Reference other files with @path/to/file — Claude reads them too (5 levels deep).
📏
Keep it Under 300 Lines
Shorter is better. Every line consumes tokens. Focus on what's actually project-specific.
🧒 ELI5 — What's CLAUDE.md for? ▼

When you start a new school year, your teacher gives you a class rulebook on day one. You don't have to explain the rules before every lesson — the teacher already knows them.

CLAUDE.md is Claude's rulebook for your project. Instead of telling Claude "we use TypeScript, never use any, always write tests" every single time — you write it once in CLAUDE.md, and Claude remembers it forever (or until you change it).
Framework

The WHAT-WHY-HOW Structure

Organize your CLAUDE.md around three questions Claude always needs to answer.

🔍
WHAT
What is this project? What tech stack? What frameworks and versions? What folder structure means what?
🧠
WHY
Why does this architecture exist? Why were these choices made? What constraints does Claude need to respect?
⚙️
HOW
How do you run dev/test/build? How do you deploy? How should Claude verify changes work?
Template

Complete CLAUDE.md Template

Copy this and adapt it to your project. Delete sections that don't apply.

markdown — CLAUDE.md
# Project: My SaaS App

## What This Is
A B2B SaaS platform for invoice management built with:
- **Frontend**: Next.js 14 (App Router), TypeScript, Tailwind CSS
- **Backend**: Node.js + Express, PostgreSQL via Prisma ORM
- **Auth**: NextAuth.js with Google OAuth
- **Infra**: Docker, deployed to Fly.io

## Project Structure
- `app/` — Next.js App Router pages and layouts
- `components/` — Shared React components
- `lib/` — Utilities, db client, auth helpers
- `prisma/` — Database schema and migrations
- `api/` — Express REST API routes (separate process)

## Important Architecture Decisions
- We use server components by default; only use `use client` for interactivity
- Never use Prisma directly in components — only in server actions or API routes
- All dates are stored as UTC in PostgreSQL; display in user's timezone client-side

## Essential Commands
```bash
npm run dev       # Start Next.js dev server (port 3000)
npm run api       # Start Express API (port 4000)
npx prisma studio # Open database GUI
npm test          # Run Jest tests
npm run lint      # ESLint + TypeScript check
```

## Code Standards
- TypeScript strict mode — no `any` types ever
- Components: PascalCase, files: kebab-case
- Use named exports (not default exports) for all components
- Write tests for all new business logic (Jest + React Testing Library)
- Commit format: `type(scope): description` (conventional commits)

## What NOT To Do
- Never bypass TypeScript with `as any` or `@ts-ignore`
- Never commit .env files or hardcode secrets
- Never modify generated Prisma client files
- Never use `npm install` without updating the lock file

## Verify Changes
After edits, always run:
1. `npm run lint` — must pass
2. `npm test` — must pass
3. `npm run build` — must succeed

See @README.md for broader project context
See @prisma/schema.prisma for the database schema

Python FastAPI Template

markdown — CLAUDE.md (FastAPI)
# Project: My FastAPI Service

## Stack
- Python 3.12, FastAPI 0.110, SQLAlchemy 2.0 (async), Alembic
- PostgreSQL (async via asyncpg), Redis (via aioredis), Pydantic v2

## Project Structure
- `app/` — FastAPI app, routers, dependencies
- `app/models/` — SQLAlchemy ORM models
- `app/schemas/` — Pydantic request/response schemas
- `app/crud/` — Database access layer (no ORM in routers)
- `alembic/` — Database migrations

## Architecture Decisions
- All DB operations in `crud/` only — never direct ORM in routers
- Use `async def` everywhere — no blocking I/O
- Pydantic v2 model_validator for complex validation (not v1 @validator)
- All IDs are UUIDs, stored as UUID type in Postgres

## Commands
```bash
uvicorn app.main:app --reload   # Dev server (port 8000)
alembic upgrade head             # Run migrations
pytest -x                        # Tests (stop on first failure)
ruff check . && mypy app/        # Lint + type check
```

## Standards
- Type hints on ALL functions — no bare `Any`
- Ruff for linting, Black for formatting (run automatically)
- Minimum 80% test coverage — use pytest-asyncio for async tests

Go Microservice Template

markdown — CLAUDE.md (Go)
# Project: User Service (Go)

## Stack
- Go 1.22, Gin v1.9, GORM v2, PostgreSQL, Redis, gRPC
- Docker, deployed to Kubernetes on GCP

## Project Layout (standard Go layout)
- `cmd/server/` — main.go entrypoint
- `internal/` — private application code
- `internal/handler/` — HTTP/gRPC handlers
- `internal/service/` — business logic
- `internal/repository/` — data access layer
- `pkg/` — shared public packages

## Critical Rules
- Never import `internal/` packages from `pkg/`
- Use interfaces for all services (enables mocking in tests)
- Context MUST be passed as first arg to all functions that do I/O
- Errors wrapped with `fmt.Errorf("...: %w", err)` — never silently dropped

## Commands
```bash
go run ./cmd/server    # Run server
go test ./...          # All tests
go vet ./...           # Static analysis
golangci-lint run      # Full lint suite
make proto             # Regenerate proto files
```
Rules

Do's and Don'ts

✅ Do Include
  • Tech stack with specific versions
  • Project folder structure with purpose of each dir
  • Commands to run dev, test, build, deploy
  • Non-obvious architecture decisions
  • Code style rules linters can't enforce
  • Domain-specific terminology
  • Git workflow and commit conventions
  • What to verify after making changes
❌ Don't Include
  • Secrets, API keys, connection strings
  • Rules linters/formatters already enforce
  • Framework basics Claude already knows
  • Historical context not relevant to work
  • Bloated explanations — be concise
  • Company secrets or internal credentials
  • Duplicate info already in README
  • Anything that could change daily
Anti-Patterns

8 Common CLAUDE.md Mistakes

#Anti-PatternWhy It's BadFix
1Teaching Claude React basicsWastes tokens on info Claude already knowsOnly document YOUR patterns, not framework basics
2Vague instruction: "Write clean code"Unactionable — Claude can't know your definition"Use named exports, no default exports, max 80 chars/line"
3Secrets in CLAUDE.mdHigh security risk — CLAUDE.md may be committedUse environment variables; reference them by name only
4Contradictory rulesClaude picks one and ignores the otherAudit for contradictions; remove or clarify conflicting rules
5CLAUDE.md over 500 linesConsumes 10k+ tokens every single sessionMove procedures to Skills; keep CLAUDE.md under 200 lines
6No "Verify Changes" sectionClaude may not run tests after editsAdd explicit verification steps: run lint → test → build
7Outdated commandsClaude runs old commands that no longer workUpdate on every stack change; include correct port numbers
8No @imports for schema/API filesClaude must guess your data structuresAdd See @prisma/schema.prisma and See @openapi.yml
Advanced

Using @imports to Stay DRY

Instead of duplicating info, reference existing files. Claude reads them in context too.

markdown — CLAUDE.md with @imports
# My Project

## Context
See @README.md for full project overview.
See @package.json for all dependencies and npm scripts.
See @prisma/schema.prisma for the database schema.
See @docs/architecture.md for architectural decisions.

## Code Standards
- TypeScript strict — no `any`
- Follow patterns in @src/components/Button.tsx as the component template
💡
@imports go 5 levels deep. Each referenced file can reference other files. But don't create deep chains — they slow down session start and consume tokens.
🧒 ELI5 — What do @imports do? ▼

Imagine your class notes reference your textbook: "See Chapter 5 for details." You don't copy the whole chapter into your notes — you just point to it.

@imports work the same way. Instead of copying your entire README into CLAUDE.md, you write See @README.md and Claude automatically reads both files. It's like a shortcut!
Maintenance

Keeping CLAUDE.md Fresh

1
Update on Every Stack Change
Every time you add a dependency, change architecture, or update a major version — update CLAUDE.md in the same commit.
2
Add a Monthly Review Task
Add a recurring task to audit CLAUDE.md. Remove stale info, update versions, trim instructions that no longer apply.
3
Let Claude Update It
Ask Claude to update CLAUDE.md when it discovers something new about the project. Claude can maintain its own context file!
bash
claude "update CLAUDE.md to reflect the migration from REST to GraphQL we just completed"
