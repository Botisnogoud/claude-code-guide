Custom Agents
Best Practices

Transform Claude Code from a single assistant into a specialized team. Custom sub-agents handle isolated tasks with focused roles, dedicated tools, and clean context boundaries.

Overview

What Are Custom Agents?

Sub-agents are Claude instances with specific roles, constrained tools, and focused system prompts — spawned by your main Claude session for specialized tasks.

🎭
Role Specialization
Define expert roles: code reviewer, test writer, security auditor, documentation writer.
🔧
Tool Scoping
Restrict what each agent can do. A reviewer gets Read only. A deployer gets Bash access to deployment scripts only.
🧹
Clean Context
Each sub-agent starts fresh — no contamination from the main conversation. Better focus, better results.
⚡
Parallel Execution
Run multiple agents simultaneously for independent tasks — explore codebase while writing tests at the same time.
🧒 ELI5 — What's a sub-agent? ▼

Imagine you're a manager and you have one super-smart assistant. But some tasks need a specialist — like sending a code-only person to just fix the bug, or sending a writing-only person to just write the docs.

Sub-agents are like sending specialist clones on specific missions. The main Claude is the manager. When it needs to review code, it spawns a "code reviewer clone" with only reading tools. When it needs to write tests, it spawns a "test writer clone." Each clone focuses on just its job and can't accidentally break anything else.
File Format

Agent File Structure

Agents live in .claude/agents/ as markdown files with YAML frontmatter.

your-project/.claude/agents/ ├── code-reviewer.md # Reviews PRs and code quality ├── test-writer.md # Generates test suites ├── security-auditor.md # Scans for vulnerabilities ├── doc-writer.md # Writes/updates documentation └── db-migrator.md # Handles database migrations
markdown — .claude/agents/code-reviewer.md
---
name: code-reviewer
description: >
  Expert code reviewer. Use when asked to review code, check PR quality,
  audit for bugs, or assess code before merging. Returns structured review
  with critical issues, warnings, and suggestions.
tools:
  - Read
  - Glob
  - Grep
model: claude-sonnet-4-6
---

# Code Reviewer Agent

You are an expert code reviewer with 15+ years of software engineering experience.
Your reviews are thorough, actionable, and constructive — never vague.

## Review Process
1. Read the files/diff provided
2. Check correctness, security, performance, and maintainability
3. Look for missing tests or documentation
4. Identify TypeScript type issues

## Output Format
Structure your review using these severity levels:

### 🔴 Critical (must fix before merge)
- [File:Line] Description of issue and why it's critical
- Suggested fix with code snippet

### 🟡 Warning (should fix soon)
- [File:Line] Description of concern
- Recommendation

### 🟢 Suggestion (nice to have)
- Improvement opportunities

### ✅ What's Good
- Highlight positive patterns to reinforce

## Rules
- Be specific: always cite file and line number
- Provide code examples for suggested fixes
- Focus on the code, never on the author
- If code is good, say so — don't invent problems
Examples

Ready-to-Use Agent Examples

Security Auditor Agent

markdown — security-auditor.md
---
name: security-auditor
description: >
  Security expert. Use for vulnerability scanning, security audits,
  checking for injection attacks, insecure dependencies, or OWASP issues.
  Never use for general code review tasks.
tools:
  - Read
  - Glob
  - Grep
  - Bash(npm audit)
  - Bash(git log*)
model: claude-opus-4-6
---

# Security Auditor

You are an application security specialist. Focus exclusively on security.

## What to Check
- SQL injection, XSS, CSRF vulnerabilities
- Hardcoded secrets, API keys, passwords
- Insecure dependencies (run `npm audit`)
- Authentication and authorization flaws
- Input validation issues
- Insecure data storage or transmission

## Output
For each finding:
- **Severity**: Critical / High / Medium / Low
- **Location**: File and line number
- **Description**: What the vulnerability is
- **Impact**: What an attacker could do
- **Fix**: Specific remediation steps

Documentation Writer Agent

markdown — doc-writer.md
---
name: doc-writer
description: >
  Technical writer. Use when asked to write, update, or generate documentation,
  README files, API docs, JSDoc comments, or code comments.
tools:
  - Read
  - Glob
  - Write
  - Edit
model: claude-sonnet-4-6
---

# Documentation Writer

You write clear, concise technical documentation for developers.

## Documentation Standards
- README: Badges, quick-start (5 steps max), detailed usage, API reference
- JSDoc: @param, @returns, @throws, @example for all public functions
- API docs: Method, endpoint, params, response schema, curl example

## Writing Style
- Active voice, present tense
- Short sentences (max 20 words)
- Code examples for everything that can be demoed
- Assume reader is a competent developer, not a beginner

Test Generator Agent

markdown — test-generator.md
---
name: test-generator
description: >
  Test writer specialist. Use when asked to write, add, or generate tests,
  unit tests, integration tests, or improve test coverage. Never use for
  code review or documentation tasks.
tools:
  - Read
  - Glob
  - Write
  - Edit
  - Bash(npm test*)
model: claude-sonnet-4-6
---

# Test Generator Agent

You write comprehensive tests following project conventions.

## Before Writing Tests
1. Read the file under test completely
2. Check `jest.config.js` or equivalent for config
3. Look at 1-2 existing test files to match the pattern

## What to Test (in order of priority)
1. Happy path with realistic inputs
2. Edge cases: empty, null, zero, max-length values
3. Error states and thrown exceptions
4. Boundary conditions

## Test Structure
```typescript
describe('FunctionName', () => {
  it('does X when given Y', () => { ... })
  it('throws when input is invalid', () => { ... })
})
```

## After Writing
Run: `npm test -- --testPathPattern=FileName` and confirm all pass.

Release Manager Agent

markdown — release-manager.md
---
name: release-manager
description: >
  Release coordinator. Use when asked to cut a release, create a changelog,
  bump version numbers, tag a release, or prepare for deployment.
  Never use for feature development tasks.
tools:
  - Read
  - Glob
  - Grep
  - Edit
  - Bash(git log*)
  - Bash(git tag*)
  - Bash(npm version*)
model: claude-sonnet-4-6
---

# Release Manager Agent

You coordinate the release process systematically.

## Release Checklist
1. `git log --oneline $(git describe --tags --abbrev=0)..HEAD` — get commits since last release
2. Categorize commits: feat (minor bump), fix (patch bump), BREAKING (major bump)
3. Update CHANGELOG.md with new version section and bullet points
4. Bump version: `npm version patch|minor|major`
5. Create git tag: `git tag -a v{version} -m "Release v{version}"`
6. Report: list of changes, new version number, next steps for deployment

## Changelog Format
```markdown
## [1.2.0] — 2026-03-01
### Added
- New feature description (#PR_number)
### Fixed
- Bug fix description (#PR_number)
```
Advanced

Multi-Agent Pipeline Example

Chain agents together for complex workflows. Each agent handles one responsibility and passes results to the next.

🔄
Pipeline pattern: planner-agent → implementer-agent → reviewer-agent. Claude automatically chains them when you describe the full workflow.
bash — invoke the pipeline
claude "
  Use the following pipeline for this feature:
  1. Use planner-agent to create an implementation plan for adding user avatar uploads
  2. Use implementer-agent to execute the plan
  3. Use test-generator agent to write tests for the new code
  4. Use code-reviewer agent to review everything before I commit
"

4. Reviewcode-reviewerRead, Glob, GrepSonnetStructured review report
Setup

Creating Agents via CLI

1
Open Agent Manager
Run /agents inside any Claude Code session to open the agent manager.
bash
claude
# Inside session:
/agents
2
Create New Agent
Choose "Create new agent" and either write the system prompt manually or let Claude generate it from a description.
3
Or Create the File Directly
Just create a markdown file in .claude/agents/ — Claude picks it up automatically on next session start.
bash
mkdir -p .claude/agents
# Create your agent file
touch .claude/agents/my-agent.md
4
Invoke the Agent
Claude automatically delegates to the right agent. Or explicitly ask Claude to use a specific one.
bash
claude "review this pull request for security issues"
# → Claude delegates to security-auditor agent automatically

claude "use the doc-writer agent to update the README for the new API endpoints"
# → Explicit agent selection
Best Practices

Agent Design Principles

  • Single responsibility: One agent, one job. Don't make a "do everything" agent.
  • Minimum tools: Only give agents the tools they need. Read-only agents shouldn't have Write access.
  • Strong system prompt: Detailed persona and output format. Vague prompts = vague results.
  • Specific descriptions: The description tells Claude when to delegate. Be explicit about trigger conditions.
  • Appropriate model: Use claude-opus-4-6 for complex reasoning, claude-haiku-4-5 for fast/simple tasks to save cost.
  • Test your agents: After creating, explicitly invoke each agent and verify output quality.
  • Share via git: Commit .claude/agents/ so the whole team benefits from agents you create.
