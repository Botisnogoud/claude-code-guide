Setup

Roll out Claude Code across your engineering team. Shared config, consistent behaviour, guardrails included.

Overview

Why Team Config Matters

Without shared config, every engineer runs Claude differently — different models, different permissions, different context. The result is inconsistent output and a tool that helps some people but not others.

🎯
Consistency
Every engineer gets the same Claude behaviour — same code style enforcement, same safety guardrails, same workflow patterns.
🔒
Security
Shared deny lists prevent Claude from running dangerous commands, committing secrets, or making destructive changes in any session.
🚀
Onboarding Speed
New engineers get full context from CLAUDE.md on day one. No manual setup — everything is in the repo and loaded automatically.
📈
Compounding ROI
Agents and skills built by one engineer benefit the whole team. Improvements compound — the team gets better at using Claude together.
🧒 ELI5 — Why does team config matter? ▼

Imagine if everyone on your team used their own version of the IDE — different settings, different linters, different keybindings. Code reviews become chaos and nobody agrees on what "correct" looks like.

Shared Claude config is like a shared .editorconfig + ESLint setup — it makes Claude behave the same for everyone. One person sets it up, everyone benefits, and Claude's output is consistent enough to trust in code review.
Configuration

Shared CLAUDE.md Strategy

Two levels of configuration work together: user-level global config and project-level config committed to the repo.

LevelLocationScopeWhat to put here
User global~/.claude/CLAUDE.mdEvery session for this userPersonal preferences, preferred tools, private API key notes
Project./CLAUDE.md (repo root)Everyone who clones the repoProject context, commands, architecture, team conventions
Subdirectory./src/CLAUDE.mdSessions inside that folderModule-specific context, micro-service conventions
📌
Merge order: Claude reads global → project → subdirectory. Later files can override or extend earlier ones. Project CLAUDE.md is the authoritative team config.

What to centralise in project CLAUDE.md

markdown — CLAUDE.md (team-optimised)
# Acme API — Team Claude Config

## Project Overview
REST API for Acme's billing platform. TypeScript + Express + PostgreSQL.
Serves ~2M requests/day. Uptime SLA: 99.9%.

## Stack
- Node.js 20, TypeScript 5.4, Express 4.18
- PostgreSQL 15 (Prisma ORM), Redis 7 (BullMQ jobs)
- Jest 29 (unit), Supertest (integration)
- pnpm workspaces

## Commands
- Dev:   `pnpm dev`
- Test:  `pnpm test`
- Lint:  `pnpm lint`
- Build: `pnpm build`
- DB:    `pnpm db:migrate` (show migration first, never auto-run)

## Team Conventions
- Branches: `feat/`, `fix/`, `chore/` prefixes
- PR titles: `feat(scope): description` (Conventional Commits)
- No `any` type — use `unknown` + type guards
- All API errors use AppError class from src/errors/

## Never Do
- Never commit without explicit engineer approval
- Never run `pnpm db:migrate` autonomously
- Never add dependencies without checking bundle impact
- Never hardcode credentials, tokens, or connection strings

## Available Agents (in .claude/agents/)
- code-reviewer: PR reviews and quality checks
- test-generator: writes Jest unit + integration tests
- security-auditor: vulnerability scanning (read-only)
- doc-writer: updates README and JSDoc

## Available Skills (in .claude/skills/)
- /review-pr: full PR review workflow
- /add-endpoint: scaffolds a new REST endpoint
- /run-tests: runs test suite and fixes failures

Security

Permission Policies

Commit .claude/settings.json to enforce shared permissions for everyone on the team. For enterprise, use managed-settings.json which cannot be overridden by users.

🏢
Enterprise: Place managed-settings.json at the org level. It takes precedence over all user and project settings.json files — users cannot override it.
json — .claude/settings.json (committed to repo)
{
  "model": "claude-sonnet-4-6",
  "smallFastModel": "claude-haiku-4-5",
  "permissions": {
    "allow": [
      "Bash(git log*)",
      "Bash(git diff*)",
      "Bash(git status*)",
      "Bash(git branch*)",
      "Bash(npm test*)",
      "Bash(npm run lint*)",
      "Bash(npm run build*)",
      "Bash(npx tsc*)"
    ],
    "deny": [
      "Bash(git push*)",
      "Bash(git reset --hard*)",
      "Bash(git rebase*)",
      "Bash(rm -rf*)",
      "Bash(curl*)",
      "Bash(wget*)",
      "Write(.env*)",
      "Write(*.pem)",
      "Write(*.key)",
      "Write(secrets/**)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/audit-bash.sh"
          }
        ]
      }
    ]
  }
}

Recommended team deny list

  • Force push: Bash(git push --force*) — prevents rewriting shared history
  • Hard reset: Bash(git reset --hard*) — prevents discarding uncommitted work
  • Recursive delete: Bash(rm -rf*) — prevents accidental mass deletion
  • Arbitrary downloads: Bash(curl*), Bash(wget*) — prevents pulling untrusted scripts
  • Env files: Write(.env*) — prevents credential exposure in commits
  • Production deploy: Bash(kubectl apply*) — deploy only via CI, never via Claude
  • Database writes: Bash(*db:migrate*) — migrations should be reviewed and run manually
Onboarding

Onboarding Checklist

10-step checklist to get a new team member productive with Claude Code on their first day.

1
Install Claude Code
Install the CLI and authenticate with the team Anthropic account or individual API key.
bash
npm install -g @anthropic-ai/claude-code
claude  # Follow auth prompts
2
Clone the repo
All team config (CLAUDE.md, .claude/) is in the repo — no separate setup needed.
bash
git clone https://github.com/your-org/your-repo
cd your-repo
# CLAUDE.md + .claude/ are already here
3
Set up global user config
Add any personal preferences to ~/.claude/settings.json — these supplement the project config.
4
Read the project CLAUDE.md
Run claude and ask: "Summarise what's in CLAUDE.md — what are the key conventions and never-do rules?"
5
Run the test suite with Claude
Verify the dev environment works end-to-end: "Run the test suite and tell me if anything fails."
6
Explore available agents
Run /agents to see the team's shared agents. Read each one to understand when to invoke it.
7
Try a skill
Pick a small task and use a team skill: "Use the review-pr skill to review my last commit."
8
Verify permissions
Confirm the deny list is active: try "run git push" — Claude should refuse and explain why.
9
Complete a real first task
Pick a small bug or chore ticket from the backlog. Use Claude to investigate, implement, and write a test.
10
Contribute back
If you improved CLAUDE.md or created a useful agent/skill, commit it. The whole team benefits immediately.
bash
git add CLAUDE.md .claude/
git commit -m "chore: improve CLAUDE.md with deploy workflow notes"
git push origin feat/improve-claude-config
CI/CD

GitHub Actions Integration

Run Claude Code in CI pipelines for automated code review, test generation, and PR feedback at scale.

🤖
Use cases: Automated PR review on every pull request, test coverage checks, security scan on merge to main, documentation freshness checks.

Automated PR review workflow

yaml — .github/workflows/claude-review.yml
name: Claude Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for git diff

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run Claude PR Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          claude --print "
            Review the diff: git diff origin/main...HEAD
            Output a concise review with:
            - Critical issues (block merge)
            - Warnings (address soon)
            - Summary verdict: APPROVE or REQUEST_CHANGES
            Keep under 500 words.
          " > review.txt
          cat review.txt

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.txt', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## Claude Code Review\n\n${review}`
            });

Security scan on merge to main

yaml — .github/workflows/claude-security.yml
name: Security Scan

on:
  push:
    branches: [main]

jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run Security Audit
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude --print "
            Scan this codebase for:
            1. Hardcoded secrets or API keys
            2. SQL injection patterns
            3. Missing input validation on user-facing endpoints
            Output ONLY Critical and High findings.
            If none found, output: CLEAN
          " > security-report.txt

      - name: Fail on Critical Findings
        run: |
          if grep -v "^CLEAN$" security-report.txt | grep -q "Critical"; then
            echo "Critical security issues found:"
            cat security-report.txt
            exit 1
          fi
          echo "Security scan passed."
🔑
API key in CI: Store ANTHROPIC_API_KEY as a GitHub Actions secret. Never hardcode it in workflow files. Use a dedicated CI API key — not a personal key — so you can rotate it independently.
Budgeting

Cost Management

Keep team API spending predictable with per-engineer budgets, usage monitoring, and smart model selection.

💳
Per-engineer API keys
Issue each engineer a separate API key from Anthropic Console. This gives per-person spend visibility and lets you revoke access instantly when someone leaves.
🔔
Spend alerts
Set monthly budget alerts in Anthropic Console. Alert at 50% and 80% of budget — catch runaway usage before it hits the limit.
📊
Usage dashboard
Anthropic Console shows daily and monthly token usage by API key. Review weekly during team standup to spot expensive workflows.
⚡
Model tiering
Default to Sonnet in settings.json. Reserve Opus for agents doing complex reasoning. Set Haiku as smallFastModel for cheap sub-tasks.

CI cost control

yaml — limit CI Claude usage
# Only run expensive Claude review on PRs to main
on:
  pull_request:
    branches: [main, staging]  # Not feature branches

# Use --max-tokens flag to cap CI spend
- name: Run Review
  run: |
    claude --print --max-tokens 2000 "
      Summarise the key changes in this PR in 3 bullet points.
      Flag any obvious bugs only.
    "
💡
Team budget ballpark: A 10-engineer team doing active Claude Code sessions typically spends $50–$200/month on Sonnet. CI usage adds $10–$50/month depending on PR volume. Set alerts accordingly.
