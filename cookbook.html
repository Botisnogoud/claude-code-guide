Recipe 1

Starting a New Project

Bootstrap any new codebase with context, conventions, and a clean first commit — all in one pass.

🚀
Use when:
Starting a greenfield project, onboarding into an existing repo for the first time, or resetting Claude's understanding of a codebase.
⚡
What you get:
A populated CLAUDE.md, sensible .gitignore, initial project structure, and a first commit with clear conventions locked in.

Recipe: /init + CLAUDE.md bootstrap

bash — run inside project root
claude
# Inside session:
/init

# Claude reads the repo and generates CLAUDE.md
# Review it, then ask Claude to fill in gaps:
"Update CLAUDE.md to include: stack details, test command,
 deploy command, code style (ESLint config), and the
 team's PR review checklist."

Why this works: /init scans the repo and writes CLAUDE.md in one shot — every future session starts with full context instead of Claude having to re-explore.

Recipe: First commit pattern

bash
claude "
  1. Scaffold a TypeScript Express project in this directory:
     - src/index.ts entry point
     - src/routes/ directory
     - src/middleware/ directory
  2. Add ESLint + Prettier config matching Airbnb style
  3. Add jest config for unit tests
  4. Write a CLAUDE.md that documents: stack, run commands,
     test command, folder structure, and coding conventions
  Do NOT commit anything — show me what you've created first.
"

Why this works: Explicit "do not commit" lets you inspect before accepting — Claude never auto-commits without permission.

Recipe: Onboarding into an existing repo

bash
claude "
  Read the entire codebase (use Glob and Read tools).
  Then write a CLAUDE.md that covers:
  - What this project does (1 paragraph)
  - Tech stack and key dependencies
  - How to run it locally
  - How to run tests
  - Folder structure (annotated tree)
  - Architecture decisions worth knowing
  - Known gotchas or non-obvious patterns
  Save to CLAUDE.md in the project root.
"

Why this works: Generates living documentation that Claude itself will use in all future sessions — every collaborator benefits too.

Recipe 2

Code Review & PR

Get a structured, actionable review of any PR or diff — faster and more consistent than waiting for a colleague.

💡
Use when: You want a second opinion before merging, you're reviewing a PR from a teammate, or you want to catch issues before pushing.

Recipe: ReviewPR skill / agent

bash
# If you have a code-reviewer agent set up:
claude "use the code-reviewer agent to review the changes
        in the current branch vs main"

# Or without an agent:
claude "
  Review the output of: git diff main...HEAD
  Structure your review with:
  - Critical issues (must fix)
  - Warnings (should fix)
  - Suggestions (nice to have)
  - What's good
  Cite file and line number for every finding.
"

Why this works: Structured severity levels force actionable output — no vague "consider refactoring this" comments.

Recipe: Multi-file diff analysis

bash
claude "
  Read these files that changed in this PR:
  - src/auth/middleware.ts
  - src/auth/token.ts
  - src/routes/user.ts
  Focus on: security issues in auth flow, type safety,
  missing error handling, and API contract consistency.
  Cross-reference how changes interact across files.
"

Why this works: Naming specific files focuses Claude on the change surface instead of re-reading the whole repo.

Recipe 3

Bug Hunting

Systematically isolate and diagnose bugs using parallel agents and git history analysis.

🐛
Use when: A bug exists but the root cause is unknown, you need to bisect a regression, or multiple components could be responsible.

Recipe: Parallel agents for diagnosis

bash
claude "
  Bug: users randomly get logged out after 10 minutes.
  Run these investigations in parallel using sub-agents:

  Agent 1 — read src/auth/ and look for token expiry
             logic, refresh token handling, and session config.

  Agent 2 — run: git log --oneline --since='2 weeks ago'
             and identify any auth-related commits.

  Agent 3 — read src/middleware/session.ts and look
             for any race conditions or missing error handling.

  Synthesize findings and propose the 3 most likely root causes
  with evidence from each agent's findings.
"

Why this works: Parallel agents finish 3x faster than sequential investigation and avoid contaminating each other's analysis.

Recipe: Git bisect pattern

bash
claude "
  Help me bisect this regression.
  - Bug: checkout flow breaks with 'undefined cart' error
  - Last known good: commit abc1234 (2 weeks ago)
  - Current broken: HEAD

  Run: git log --oneline abc1234..HEAD
  Then identify which commits touch src/cart/ or src/checkout/.
  For each suspect commit, describe what changed and
  why it might cause 'undefined cart'.
  Rank suspects by likelihood.
"

Why this works: Claude reads the commit history and correlates changes to the symptom — narrows bisect from 50 commits to 2-3 suspects in seconds.

Recipe 4

Refactoring at Scale

Use worktree isolation to refactor safely — changes stay isolated until you're ready to merge.

🌿
Use when: Renaming across 30+ files, restructuring a module, or making a breaking API change that touches many call sites.

Recipe: Worktree isolation for safe refactoring

bash
# Create isolated worktree for the refactor
git worktree add ../my-project-refactor -b refactor/rename-user-to-account

# Open Claude in the worktree
cd ../my-project-refactor
claude "
  Rename all occurrences of 'User' → 'Account' across this codebase.
  Rules:
  - Rename TypeScript types, interfaces, class names, variable names
  - Update import paths if files are renamed
  - Do NOT rename in test fixture data or migration files
  - After renaming, run: npx tsc --noEmit to check for type errors
  Show me a summary of all files changed before writing anything.
"

Why this works: Worktree keeps the refactor off your main branch — you can run tests, inspect, and merge only when confident.

Recipe: Batch restructure with plan-first pattern

bash
claude "
  I want to restructure src/ from a flat layout to
  feature-based folders.

  Step 1: Read the current src/ structure with Glob.
  Step 2: Propose the new folder structure — show me
          the before/after mapping as a table.
  Step 3: STOP and wait for my approval before moving
          any files.

  Do not touch anything until I say 'proceed'.
"

Why this works: Plan-first with an explicit stop gate prevents Claude from restructuring 40 files in a direction you didn't intend.

Recipe 5

Writing & Fixing Tests

Generate comprehensive test suites and fix flaky tests without touching production code.

🧪
Use when: Coverage is too low, a new feature needs tests, or CI is failing due to flaky tests you need to diagnose.

Recipe: Test-runner agent pattern

bash
claude "
  Write unit tests for src/cart/cart.service.ts.
  Before writing:
  1. Read the file completely
  2. Read jest.config.js for test setup
  3. Read one existing test file to match the pattern

  Then write tests covering:
  - All public methods (happy path)
  - Edge cases: empty cart, negative quantity, null item
  - Error states: item not found, cart limit exceeded

  After writing, run: npm test -- --testPathPattern=cart.service
  and fix any failures before handing back to me.
"

Why this works: Reading existing tests first ensures new tests match your project's exact patterns — no style mismatch to clean up.

Recipe: Coverage analysis and gap fill

bash
claude "
  Run: npm test -- --coverage --coverageReporters=text
  Read the coverage output.
  Identify the 5 files with lowest coverage that are
  NOT in: migrations/, generated/, or vendor/.
  For each file, write tests to bring coverage above 80%.
  Run coverage again after each file to confirm improvement.
"

Why this works: Targeting the lowest-coverage files gives maximum ROI per token spent — not random test generation.

Recipe 6

Database Migrations

Generate, review, and validate database migrations safely — with a mandatory human review step before execution.

⚠️
Always include a review gate. Never let Claude execute migrations autonomously. Generate → review → execute manually.

Recipe: Safe migration with review step

bash
claude "
  I need to add a 'last_login_at' timestamp column to
  the users table.

  1. Read existing migrations in db/migrations/ to match format
  2. Generate the migration file (up + down)
  3. Check if any TypeScript models reference the users table
     and update them to include the new field
  4. Show me everything you've created — DO NOT run the migration.

  I will review and run: npm run db:migrate manually.
"

Why this works: Explicit "do not run" + manual execution step ensures no accidental schema changes on a production-connected dev database.

Recipe: Migration audit for rollback safety

bash
claude "
  Read all migration files in db/migrations/.
  For each migration that lacks a proper 'down' function,
  list it with: filename, what the 'up' does, and what
  a correct 'down' rollback should look like.
  Do not modify any files — just give me the audit report.
"

Why this works: Read-only audit finds rollback gaps before they become production incidents — no changes, just actionable findings.

Recipe 7

API Integration

Scaffold new endpoints and API clients with full type safety — matching your existing patterns exactly.

🔌
Use when: Adding a new REST endpoint, integrating a third-party API, or adding a new service client with proper error handling.

Recipe: Scaffold new endpoint from existing pattern

bash
claude "
  Add a PATCH /users/:id/preferences endpoint.
  Before writing anything:
  1. Read src/routes/users.ts — match its exact pattern
  2. Read src/controllers/user.controller.ts — match its style
  3. Read src/types/user.types.ts — extend types here

  Then create:
  - Route handler in src/routes/users.ts
  - Controller method in src/controllers/user.controller.ts
  - Request/response types in src/types/user.types.ts
  - Unit test for the controller method

  Follow all existing error handling patterns exactly.
"

Why this works: Reading existing patterns first guarantees the new endpoint is indistinguishable from hand-written code — no style cleanup needed.

Recipe: Third-party API client with types

bash
claude "
  Integrate the Stripe API for subscription billing.
  1. Read package.json — check if stripe SDK is already installed
  2. Create src/services/stripe.service.ts with:
     - Full TypeScript types for all request/response shapes
     - createSubscription, cancelSubscription, getInvoices methods
     - Proper error handling (StripeError types)
     - Unit tests with mocked stripe calls
  3. Add STRIPE_SECRET_KEY to .env.example (not .env)
  Do not hardcode any credentials.
"

Why this works: Explicit "add to .env.example not .env" prevents accidental credential commits — a common pitfall when Claude scaffolds integrations.

Recipe 8

Security Audit

Run a read-only security scan to find vulnerabilities without touching a line of production code.

🔒
Use when: Before a major release, after adding new dependencies, or when integrating user-generated input into a new part of the system.

Recipe: Read-only security scan agent

bash
claude "
  Perform a security audit. You are in READ-ONLY mode —
  do not modify any files.

  Check for:
  1. Hardcoded secrets, API keys, or passwords (grep for common patterns)
  2. SQL injection risks (raw query construction)
  3. XSS vulnerabilities (unescaped user input rendered in HTML)
  4. Missing input validation on API endpoints
  5. Insecure direct object references (IDOR)
  6. Run: npm audit and report HIGH/CRITICAL findings

  Output: severity-sorted findings with file, line, description,
  and a specific remediation for each.
"

Why this works: Explicit read-only constraint prevents Claude from "fixing" vulnerabilities before you've reviewed them — audit first, fix second.

Recipe: Dependency vulnerability check

bash
claude "
  Run: npm audit --json
  Parse the output and:
  1. List all CRITICAL and HIGH vulnerabilities
  2. For each: package name, CVE, what it allows an attacker to do,
     and the safe version to upgrade to
  3. Check if any vulnerable packages are in our direct dependencies
     (not just transitive) by reading package.json
  4. Propose the minimal set of version bumps to fix all criticals
  Do not run npm install — just give me the upgrade plan.
"

Why this works: Audit-then-plan pattern gives you a reviewable upgrade list — you stay in control of which deps get bumped.

Recipe 9

Documentation Generation

Generate accurate, up-to-date documentation by letting Claude read the actual code — not guess from summaries.

📝
Use when: README is stale, JSDoc comments are missing, or you need an API reference for a new service before launch.

Recipe: Doc-writer skill pattern

bash
claude "
  Use the doc-writer agent to update README.md.
  The README should include:
  - Project description (what it does, who it's for)
  - Prerequisites and installation (exact commands)
  - Quick start (working in under 5 minutes)
  - Environment variables table (name, required, default, description)
  - API reference (auto-generated from src/routes/)
  - Contributing guide

  Read the actual source code — do not make up API endpoints.
  Preserve any existing sections that are still accurate.
"

Why this works: "Read the actual source" instruction prevents hallucinated endpoint docs — a common failure mode for LLM documentation.

Recipe: JSDoc for all public functions

bash
claude "
  Add JSDoc comments to all exported functions in src/services/.
  For each function:
  - @param with type and description for every parameter
  - @returns with type and description
  - @throws for any errors thrown
  - @example with a realistic usage example

  Read each file first, then add comments inline.
  Do not change any logic — comments only.
  Run: npx tsc --noEmit after to confirm no type errors introduced.
"

Why this works: "Comments only, no logic changes" + TypeScript compile check prevents documentation from accidentally mutating behaviour.

Recipe 10

Team Onboarding

Give every new team member instant productivity by sharing CLAUDE.md and skills via the repo.

👥
Use when: Onboarding a new engineer, sharing Claude workflows across your team, or standardising how your team uses Claude Code.

Recipe: Shared CLAUDE.md + shared skills setup

bash
# Commit both CLAUDE.md and .claude/ to the repo
git add CLAUDE.md .claude/
git commit -m "chore: add Claude Code team configuration"

# New team member gets everything on clone:
git clone https://github.com/your-org/your-repo
cd your-repo
claude  # CLAUDE.md + all agents + all skills load automatically

Why this works: Claude loads CLAUDE.md and .claude/ automatically on session start — zero setup for new team members.

Recipe: Generate team onboarding doc for Claude

bash
claude "
  Read the codebase and write a CLAUDE.md optimised for
  onboarding a new engineer joining our team.
  Include:
  - What the project does (for someone with zero context)
  - Development environment setup (step by step)
  - The 3 most common tasks and how to do them with Claude
  - Which agents/skills we have and when to use each
  - Team conventions (branching, PR titles, commit format)
  - Things Claude should NEVER do in this project
  Save to CLAUDE.md.
"

Why this works: A CLAUDE.md written for a new engineer also perfectly primes Claude — clear context, explicit constraints, workflow instructions.
